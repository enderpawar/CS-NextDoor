# Velog 포스팅 작성

## 사전 준비 (자동 수행)

아래 순서로 오늘 작업 내용을 수집합니다.

1. **세션에서 다룬 개념·파일·오류** 목록화
2. **트러블슈팅 흐름** 재구성 (문제 → 원인 → 해결)
3. **핵심 인사이트** 추출 — 단순 구현 나열이 아닌 "왜 이렇게 했는가"

인자가 있으면 해당 키워드 중심으로 좁혀서 작성합니다: **$ARGUMENTS**

---

## 글 구조

아래 섹션을 순서대로 작성합니다. 분량은 실제 작업 복잡도에 맞게 자연스럽게 조절하세요.

각 섹션은 반드시 `##` 마크다운 헤딩으로 시작해야 합니다 (번호 목록 금지).

- `## 들어가며`
- `## 배경 / 왜 이게 필요했나`
- `## 핵심 개념 정리`
- `## 구현 과정`
- `## 트러블슈팅`
- `## 마치며`

---

## 문체 & 톤 가이드

### 기본 원칙

- **존댓말** 사용. 단, 딱딱하지 않게 — "~합니다" 보다 "~해요", "~거든요" 병용 가능
- 독자는 **비슷한 스택을 다루는 개발자** — 기초 설명은 짧게, 핵심에 집중
- 코드 블록은 **실제 작성한 코드** 그대로. 주석으로 포인트 강조
- 트러블슈팅은 **삽질 과정을 솔직하게** — "이게 왜 안 되지?" 감정선도 OK

### 금지 사항

- "안녕하세요, 오늘은 ~에 대해 알아보겠습니다" 류 형식적 인트로 ❌
- 공식 문서 그대로 복붙한 개념 설명 ❌
- 결론 없는 트러블슈팅 나열 ❌
- 과도한 강조 (볼드, 이탤릭 남발) ❌

### 좋은 예시 인트로

> Claude Code 설정을 정비하다가 `jq`가 Windows에 없다는 걸 뒤늦게 알았어요.
> PowerShell로 JSON을 파싱하는 게 이렇게 번거로운 줄은 몰랐는데,
> 덕분에 `ConvertFrom-Json`이랑 친해졌습니다.

## 각 섹션 작성 지침

### 1. 들어가며
- 오늘 작업의 **맥락과 동기**를 2~4문장으로
- "어떤 문제를 풀려다가 이걸 하게 됐는지" 흐름

### 2. 배경 / 왜 이게 필요했나
- 기술 선택의 **이유** (대안과 비교 포함)
- 이 섹션이 없어도 될 만큼 간단하면 생략

### 3. 핵심 개념 정리
- 오늘 새로 알게 된 / 제대로 이해한 개념
- 개념당 **코드 예시 1개** + 한 줄 설명
- 공식 문서 링크 대신 **직접 설명**으로

### 구현 과정
- 전체 흐름을 단계별로 (1. 2. 3. 순서)
- 핵심 코드 블록 포함, 불필요한 보일러플레이트는 생략
- "이 부분이 왜 이렇게 생겼는지" 설명 중심

### 4. 트러블슈팅
형식:
```
**문제**: ~했더니 ~가 발생했습니다
**원인**: ~이었어요 / ~를 놓쳤습니다
**해결**: ~로 변경했고, ~를 확인했습니다

// 변경 전
코드

// 변경 후
코드
```
- 오류 메시지가 있으면 그대로 인용
- "왜 이게 원인이었는지" 설명이 핵심

### 마치며
- 오늘 작업의 **핵심 교훈** 1~3개 (글머리 기호)
- 다음에 이어서 할 것 / 남은 과제 (있으면)
- 2~3문장으로 마무리

## 출력 형식

Velog에 바로 붙여넣을 수 있는 **Markdown** 형식으로 출력해 루트 폴더에 md파일로 저장합니다.

```markdown
---
title: [제목]
tags: [태그1, 태그2, 태그3]
---

## 들어가며

[본문]

## 배경 / 왜 이게 필요했나

[본문]

## 핵심 개념 정리

[본문]

## 구현 과정

[본문]

## 트러블슈팅

[본문]

## 마치며

[본문]
```

> **중요**: 각 섹션 제목은 반드시 `##`로 시작하는 마크다운 헤딩으로 작성합니다. Velog는 `##` 헤딩을 기반으로 목차를 자동 생성합니다.

태그는 오늘 다룬 기술 스택 기준으로 5개 이내로 추천합니다.
제목은 "~하기" / "~를 제대로 이해해보자" 스타일로.
